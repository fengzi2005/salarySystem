## 一、需求确认阶段

### Q1: 数据库表结构谁来设计？
**答**：由我自行设计。共设计15张表，覆盖组织、人事、考勤、薪资、权限、日志六大领域。

### Q2: 基本工资和岗位工资是什么关系？
**答**：基本工资 ≠ 岗位工资，二者完全独立、分开核算。基本工资是员工个人专属基础底薪（存在employee表），岗位工资是和岗位绑定的统一标准（存在position表）。职级工资同理（存在title表）。

### Q3: Spring Boot 版本选哪个？
**答**：从4.1.0 pre-release降级到3.4.1稳定版。同时补全缺失依赖（web、mybatis-plus、redis、jjwt、aop、validation）。

### Q4: MySQL密码到底是多少？
**答**：原文档写123456，实际是123。后来发现Redis密码确实是123456。

### Q5: UI风格怎么定？
**答**：左侧导航 + 右侧内容区的经典后台布局。蓝白配色，左侧从浅蓝到深蓝渐变。佛山大学logo放在左侧导航顶部。

## 二、数据库核心设计对话

### 工资计算逻辑讨论
```
用户：自动结算时有没有考虑该员工入职日期与当前选择月份的关系？
AI：没有。当前只过滤了在职员工，没判断入职日期是否在所选月份之前。
    → 修复：游标增加条件 AND e.entry_date <= LAST_DAY(当月)
```

### 存储过程 NOT FOUND 陷阱（最严重的Bug）
```
用户：数据展示不出来，只有一条记录
AI：内层SELECT INTO考勤数据时，找不到记录触发NOT FOUND，
    连外层的CONTINUE HANDLER也触发了，游标直接退出。
    → 修复：改为SET变量 = (SELECT ...) 子查询方式
```

### 工龄工资规则是否真正参与计算？
```
用户：工龄工资规则有无真正参与工资计算？
AI：参与了。测试验证：
    旧规则(每年+300)：王五6年 × 300 = 1800元 ✓
    新规则(第2年+100)：王五 300 + 5×100 = 800元 ✓
    需删除旧记录全新计算才能看到差异（因CONFIRMED记录被跳过）
```

### 工资项配置启停联动
```
用户：事假扣款停用后，自动计算还是扣了
AI：存储过程没读取 salary_item_config → 增加12个is_active判断，
    计算前逐项检查，停用项直接置零
```

### 已确认工资被覆盖
```
用户：大部分已确认，两个草稿，点自动计算会影响已确认的吗？
AI：会的，这是bug。→ 增加 IF NOT EXISTS(status='CONFIRMED') 检查，
    已确认记录跳过不更新
```

### 工资项配置编码列
```
用户：编码这一列的意义是什么？
AI：BASE_SALARY、PROJECT_BONUS这种，给存储过程识别工资项用的
用户：删掉
AI：已删除
```

## 三、权限系统设计对话

### 第一版：简单角色权限
```
用户：个人只能查看自己的工资，admin可以查看全部员工的工资
AI：实现三级角色 + 路由守卫 + 侧边栏过滤
```

### 第二版：岗位级别权限
```
用户：分部经理只能管理分部，部门经理只能管理所在一级部门及其子部门，
      总经理能够管理所在总部及其总部门
AI：实现四级岗位级别：总经理(4) > 副总经理(3) > 部门经理(2) > 分部经理(1)
    管理岗可管：技术岗 + 下级管理岗
    不能管：同级或上级管理岗
```

### 权限规则的三次修正
```
第一版：AI按role_code判断 → MANAGER角色全都能管
第二版：增加position_name判断 → 四经理可管，但范围太宽
第三版（最终）：
  用户："姚铮作为在开发部门的分部经理，只能管开发部门旗下所有分部"
  用户："首先每个经理可能管理的部门范围是 自己所在的部门及其子部门，
         在这些部门中，级别比自己高的部门不能管"
  AI：findScopeRootId改为以自己部门为根，配合getAllowedLevels限制层级
```

### 部门删除权限的简化
```
用户：算了，部门管理展示所有部门，但不是系统管理员都不给编辑和新增，按钮变灰就可
AI：回退后端canManageDept检查，改为前端按钮disabled控制
```

## 对话特征总结

| 类型 | 占比 | 示例 |
|------|:--:|------|
| 功能开发 | 40% | "给所有实体管理页面加分页条" |
| UI调整 | 30% | "字体调大"、"颜色深一点"、"往左一点" |
| Bug修复 | 20% | "没数据"、"报错了"、"权限没拦住" |
| 需求确认 | 10% | "基本工资和岗位工资是什么关系" |

**沟通特点**：
- 用户往往先描述现象，结合自己掌握的知识给出问题所在，AI自行校对正误后改正
- UI调整需要多次微调（字体、间距、颜色）
- 复杂权限规则需要多轮讨论才能精确锁定
- 字段命名不一致是最高频的Bug来源（camelCase vs snake_case累计出现5+次）
